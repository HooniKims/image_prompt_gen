# 이미지 프롬프트 확장 프로그램 개발기

초보 교사 개발자를 위한 Human in the Loop 캡스톤 프로젝트 서사

## 1. 시작은 코드가 아니라 불편함이었다

처음부터 “크롬 확장 프로그램을 만들자”는 생각으로 출발한 것은 아니었다.

출발점은 훨씬 더 교실에 가까웠다.

수업 자료를 만들 때 이미지 생성 AI는 분명 유용했다. 과학 개념을 그림으로 설명하거나, 역사 장면을 시각화하거나, 학생용 활동지를 더 친근하게 만들 수 있었다. 그런데 막상 사용해 보면 매번 같은 벽에 부딪혔다.

“프롬프트를 어떻게 써야 하지?”

교사는 수업 목표를 알고 있다. 학생 수준도 알고 있다. 어떤 오개념이 생길 수 있는지도 안다. 하지만 이미지 생성 모델이 좋아하는 언어로 그것을 정리하는 일은 또 다른 기술이었다.

예를 들어 “초등학생에게 물의 순환을 설명하는 그림”이라고만 입력하면 결과가 애매해질 수 있다. 그림은 예쁘지만 수업에 바로 쓰기 어렵거나, 한글 문구가 깨지거나, 필요한 단계가 빠질 수 있다.

그래서 질문이 생겼다.

“교사가 수업 의도만 입력하면, 이미지 생성 AI가 이해하기 좋은 프롬프트로 바꿔주는 도구를 만들 수 없을까?”

이 질문이 프로젝트의 시작이었다.

## 2. 생각과 사고가 먼저였다

초보 개발자가 가장 자주 빠지는 함정은 바로 코드를 먼저 치는 것이다.

버튼부터 만들고, 입력창부터 만들고, API부터 붙인다. 그런데 그렇게 하면 어느 순간 이런 질문이 돌아온다.

“이 기능은 왜 필요한가?”

이번 프로젝트에서는 순서를 바꿨다.

먼저 생각했다.

- 교사는 어떤 상황에서 이 도구를 쓰는가?
- 이미지 프롬프트 작성에서 가장 귀찮은 부분은 무엇인가?
- 초보 교사 개발자도 이해할 수 있는 구조여야 한다면 어디까지 단순해야 하는가?
- ChatGPT와 Gemini를 모두 쓰려면 사용 흐름은 어떻게 같아야 하는가?
- 서버나 API가 실패했을 때 사용자가 원인을 알 수 있어야 하지 않는가?

이 단계에서 중요한 것은 “정답을 한 번에 맞히는 것”이 아니었다. 중요한 것은 생각을 밖으로 꺼내는 일이었다.

저와 AI는 계속 티키타카를 했다.

처음에는 단순한 프롬프트 변환기처럼 생각했다. 그런데 대화를 하다 보니 교실에서는 단순 이미지 한 장만 필요한 것이 아니었다.

스토리북이 필요할 때도 있고, 만화 컷이 필요할 때도 있고, 여러 장의 시리즈 이미지가 필요할 때도 있었다. 학부모 안내용 이미지와 학생 활동용 이미지는 톤도 달라야 했다.

이때 AI는 코드를 대신 짜는 도구가 아니라, 생각을 밀어붙이는 파트너가 되었다.

“이 기능이 정말 필요한가?”

“사용자는 이걸 어디서 누를까?”

“초보자가 이해할 수 있을 만큼 단순한가?”

“한 장 생성과 여러 장 생성은 같은 폼으로 처리할 수 있는가?”

이 질문들이 쌓이면서 제품의 뼈대가 만들어졌다.

## 3. Human in the Loop: 사람이 방향을 잡고 AI가 밀어준다

이 프로젝트의 핵심은 Human in the Loop였다.

여기서 Human in the Loop는 단순히 “사람이 중간에 검토한다”는 뜻이 아니다. 사람이 방향을 정하고, AI가 제안을 하고, 다시 사람이 판단하고, 다시 AI가 구현하는 순환 구조를 말한다.

이번 작업에서도 그 구조가 계속 반복되었다.

제가 먼저 요구를 던졌다.

“ChatGPT와 Gemini에서 바로 쓸 수 있게 만들자.”

AI가 구조를 제안했다.

“크롬 확장 프로그램으로 만들고, 오른쪽 사이드 패널을 주입하면 됩니다.”

제가 다시 수정했다.

“초보자가 써야 하니까 설정은 숨기고, 기본 흐름은 간단해야 한다.”

AI가 구현을 바꿨다.

“기본 API 주소를 넣고, 관리자 설정은 접어두겠습니다.”

다시 제가 문제를 발견했다.

“스타일이 너무 많다. 50개를 그대로 보여주면 초보자는 못 고른다.”

AI가 처음에는 단순히 앞의 8개만 보여주는 방식으로 갔다.

그러자 제가 다시 잡았다.

“처음 8개가 아니라, 비슷한 것들을 카테고리화해서 대표 8개를 추려야 한다.”

이 지점이 중요하다.

AI는 빠르게 만들 수 있지만, 맥락을 놓칠 수 있다. 사람은 느리지만, 목적과 맥락을 붙잡을 수 있다. 둘이 같이 갈 때 결과가 좋아진다.

결국 스타일은 49개를 그대로 나열하지 않고 8개 카테고리로 묶었다.

- 웹/UI·SaaS
- 교육·설명
- 카드뉴스·에디토리얼
- 데이터·비교·컨설팅
- 일러스트·공예
- 레트로·코믹·팝
- 3D·아이소메트릭
- 시네마틱·미래형

이것이 Human in the Loop의 실제 모습이다.

사람이 “아니야, 처음 8개가 아니라 대표성이 있어야 해”라고 말하는 순간, 제품의 수준이 올라간다.

## 4. 캡스톤 프로젝트처럼 문제를 제품으로 수렴시켰다

이 작업은 작은 기능 하나를 만든 것이 아니라, 하나의 캡스톤 프로젝트에 가까웠다.

캡스톤 프로젝트는 단순 과제가 아니다. 실제 문제를 정의하고, 사용자 흐름을 설계하고, 작동하는 산출물로 끝까지 만드는 과정이다.

이 프로젝트도 같은 흐름을 가졌다.

첫째, 문제를 정의했다.

교사가 이미지 생성 AI를 수업에 쓰고 싶지만, 좋은 프롬프트 작성이 어렵다.

둘째, 사용자를 정의했다.

대상 사용자는 전문 개발자가 아니라, AI 도구를 수업과 자료 제작에 활용하려는 교사다.

셋째, 사용 맥락을 정의했다.

교사는 ChatGPT나 Gemini 화면에서 이미 작업 중이다. 따라서 별도 웹사이트를 열게 하기보다, 현재 화면 오른쪽에서 바로 도와주는 편이 자연스럽다.

넷째, 시스템 구조를 정했다.

크롬 확장 프로그램이 사용자 입력을 받고, hosted Prompt API가 Mac mini의 LM Studio 모델을 호출해 좋은 프롬프트를 만든다. 일반 사용자는 LM Studio 주소를 몰라도 된다. 확장은 기본적으로 `https://image-prompt.alluser.site/api/image-prompt`에 자동 연결되고, API 호출은 content script가 아니라 background service worker가 맡는다.

다섯째, 검증했다.

실제로 ChatGPT 페이지에서 프롬프트를 넣어 보고, 이미지 생성까지 확인했다. UI 캡처를 보내고, ZIP 파일을 만들고, 최종 스펙 문서까지 정리했다.

이 과정 전체가 하나의 캡스톤이었다.

중요한 것은 “완벽한 코드를 한 번에 짜는 것”이 아니라, 문제에서 출발해 제품으로 수렴하는 경험이다.

## 5. 기능은 대화 속에서 자랐다

처음 기능 목록은 단순했다.

- 주제 입력
- 스타일 선택
- 비율 선택
- 프롬프트 생성
- 입력창에 삽입

하지만 대화가 이어지면서 기능은 점점 교실에 가까워졌다.

“스토리북도 필요하지 않을까?”

“만화나 웹툰 컷도 만들 수 있어야 하지 않을까?”

“여러 장을 한 번에 생성하려면 numbered prompt pack이 필요하지 않을까?”

“학생용과 학부모용 안내 이미지는 말투와 구성 자체가 다르지 않을까?”

이 질문들이 폼 기능으로 바뀌었다.

결국 확장 프로그램은 단순 프롬프트 생성기가 아니라, 여러 작업 유형을 가진 작은 제작 도구가 되었다.

사용자는 작업 유형을 고르고, 생성 장수와 장면 구성을 입력하고, 캐릭터와 배경 일관성 조건을 넣을 수 있다.

예를 들어 스토리북을 만들 때는 “1장: 주인공이 문제를 만난다, 2장: 친구와 함께 해결한다, 3장: 배운 점을 정리한다”처럼 장면 계획을 넣을 수 있다.

만화라면 컷 구성과 말풍선 최소화 조건을 줄 수 있다.

나중에는 “스토리북이나 만화라면 사용자가 말하지 않아도 대사가 들어가야 하지 않을까?”라는 질문도 나왔다. 실제로 스토리북과 웹툰은 장면만 있는 이미지보다 짧은 대사나 말풍선이 있을 때 의도가 더 잘 전달된다. 그래서 기본값을 바꾸었다. 스토리북은 페이지마다 짧은 대사나 캡션을, 만화/웹툰은 컷마다 간결한 말풍선을 넣도록 지시한다. 다만 이미지 생성 모델은 글자 오류가 생길 수 있으므로 문구는 짧고 렌더링하기 쉽게 제한한다. 사용자가 `대사 없음`, `말풍선 없음`, `텍스트 최소화`처럼 말하면 그 조건을 우선해 기본 대사 지시는 빼도록 했다.

여러 장 시리즈라면 같은 캐릭터, 같은 색감, 같은 배경 톤을 유지하라고 지시할 수 있다.

나중에는 더 현실적인 사용 문제가 보였다. 사용자가 생성 장수만 8장이나 10장으로 고르고 장면/컷 구성을 비워 두면, 모델이 여러 장을 만들어야 한다는 의도를 약하게 받아들일 수 있었다. 그래서 장면 계획을 쓰지 않아도 선택한 장수만큼 정확히 번호가 붙은 프롬프트팩을 만들고, 원본 아이디어를 자연스러운 장면이나 컷으로 직접 나누라는 지시를 추가했다. 생성 장수 선택지도 중간 숫자를 건너뛰지 않고 1장부터 12장까지 1장 단위로 고를 수 있게 바꾸었다.

이런 기능은 처음부터 나온 것이 아니다.

대화 속에서 발견되었다.

초보 교사 개발자에게 중요한 메시지는 이것이다.

기능은 혼자 머릿속에서 완성되지 않는다. 사용자와 대화하고, AI와 대화하고, 결과를 보면서 다시 질문할 때 자란다.

## 6. 좋은 프롬프트는 감이 아니라 기준에서 나온다

중간에 중요한 질문이 나왔다.

“Image 2.0, Nano Banana 가이드 문서 반영은?”

이 질문은 단순히 참고 자료를 넣었냐는 확인이 아니었다. 실제 프롬프트 생성에 그 기준이 반영되는지를 묻는 질문이었다.

그래서 레퍼런스 구조를 분명히 했다.

OpenAI의 이미지 생성 가이드와 Google DeepMind의 Gemini Image / Nano Banana 가이드를 프로젝트 안에 요약 문서로 정리했다.

처음에는 별도 레퍼런스 문서로 정리하는 방식도 생각했지만, 최종 확장 프로그램 폴더에서는 실행 경로가 더 중요했다. 그래서 중요한 기준을 코드에 직접 반영했다.

```text
src/content-runtime.js
```

핵심은 `IMAGE_GUIDE_SET`, `canonicalGuideSet()`, `targetGuideInstructions()`였다.

쉽게 말하면 이렇다.

공식 가이드를 매번 인터넷에서 읽어 오는 것이 아니다. 공식 문서에서 중요한 규칙을 뽑아 제품 안의 런타임 기준으로 넣어 두고, 프롬프트를 만들 때마다 그 규칙이 자동으로 섞이게 했다.

이 방식은 안정적이다.

서버가 살아 있을 때는 LM Studio / OpenAI-compatible system prompt에 같은 기준이 들어간다.

비상용 브라우저 fallback 코드에도 같은 기준이 남아 있지만, 정상 생성 버튼은 API 응답이 있을 때만 성공으로 처리한다. API가 실패했는데도 임시 프롬프트를 성공처럼 보여 주는 방식은 사용자에게 오해를 줄 수 있기 때문이다.

즉, 사용자는 “왜 어떤 때는 잘 되고 어떤 때는 이상하지?”라는 혼란을 덜 겪게 된다.

초보 개발자에게 여기서 배울 점은 분명하다.

좋은 AI 결과물은 감으로만 나오지 않는다. 기준을 정하고, 그 기준이 실제 실행 경로에 들어가야 한다.

## 7. 저작권과 용량도 제품의 일부다

처음에는 스타일 썸네일로 외부 샘플 이미지를 쓸 수 있었다. 눈으로 보기에는 편했다. 하지만 마음에 걸리는 지점이 있었다.

“이 썸네일들을 그대로 써도 괜찮을까?”

이 질문도 Human in the Loop의 중요한 장면이다.

AI가 기능을 만들었다고 끝이 아니다. 사람이 윤리와 운영을 검토해야 한다.

결국 외부 원본 이미지를 ZIP에서 제거했다. 대신 image-gen으로 자체 생성한 스타일 썸네일을 만들었다. 처음에는 8개 카테고리 대표 썸네일만 있었지만, 실제 사용해 보니 여러 스타일이 같은 이미지를 공유해서 개성이 약했다. 그래서 49개 스타일 각각에 맞는 썸네일을 다시 만들고, `style-01.webp`부터 `style-49.webp`까지 개별 파일로 연결했다. 한 번 더 확인해 보니 1:1 이미지를 16:9 슬롯에 넣고 남는 공간을 블러 처리한 흔적이 보였다. 이것도 실제 UI에는 어울리지 않았다. 결국 처음부터 가로형 16:9 구도에 맞춘 시트로 다시 만들었고, 칠판 스타일은 초록 칠판과 흰 분필 선이 보이도록 고쳤다.

확장 프로그램 아이콘도 빠져 있었다. 그래서 image-gen으로 브라우저 확장 프로그램에 어울리는 아이콘을 만들고, Chrome이 요구하는 16/32/48/128 PNG 크기로 저장했다. 작은 아이콘에서도 보이도록 이미지 프레임, 프롬프트 커서, 보조 도구의 느낌을 단순한 앱 아이콘 형태로 정리했다.

하지만 또 다른 문제가 있었다.

“확장 프로그램 용량이 너무 커지면 안 된다.”

그래서 썸네일을 고화질 이미지로 넣지 않았다. 느낌만 전달되면 충분했다.

최종적으로 WebP로 압축했다. 이후 확장 아이콘 4종이 추가되면서 ZIP 크기는 커졌지만, 작업용 원본 아이콘 이미지와 미리보기용 시트는 ZIP에서 제외해 배포 용량을 다시 줄였다.

이 장면에서 배울 수 있는 것은 단순하다.

제품 개발은 기능만 만드는 일이 아니다.

권리, 용량, 배포, 유지보수까지 함께 생각해야 한다.

## 8. 모두가 바로 쓸 수 있는 API 구조

이 프로젝트에는 Mac mini Prompt API가 있다.

Chrome 확장 프로그램은 기본적으로 아래 주소를 호출한다.

```text
https://image-prompt.alluser.site/api/image-prompt
```

이 서버는 LM Studio의 OpenAI-compatible API와 연결되어 있다. 기본 모델은 `google/gemma-4-e2b`, 복잡한 요청용 모델은 `google/gemma-4-e4b`다.

중요한 전환점은 “내 컴퓨터에서만 되는 구조”를 버린 것이다.

처음에는 로컬 LM Studio 주소를 직접 넣어 테스트할 수 있었다. 하지만 이 도구는 나만 쓰는 것이 아니라 다른 사람도 써야 한다. 사용자가 `localhost`나 LM Studio 프록시 주소를 직접 입력해야 한다면 설치 경험이 복잡해진다.

그래서 기본 배포판은 hosted Prompt API에 자동으로 연결되도록 정리했다. 예전 테스트 과정에서 저장된 `lm.alluser.site`, `localhost`, `127.0.0.1` 같은 직접 호출 주소가 남아 있어도 확장 초기화 시 기본 API 주소로 되돌린다.

또 하나의 문제는 호출 위치였다. ChatGPT나 Gemini 페이지 안에 주입된 content script가 외부 API를 직접 호출하면 CORS나 origin 제한에 막힐 수 있다. 그래서 API 호출은 background service worker로 옮겼다.

이제 흐름은 이렇다.

- 사용자가 프롬프트 생성을 누른다.
- content script가 background service worker에 요청을 보낸다.
- background service worker가 hosted Prompt API를 호출한다.
- hosted Prompt API가 Mac mini / LM Studio에 요청을 전달한다.
- 일반 요청은 `google/gemma-4-e2b`로 호출된다.
- 품질이 높음이거나 여러 장 생성, 스토리북/웹툰/시리즈, 장면 구성, 일관성 조건, 이미지 안 문구가 있는 요청은 `google/gemma-4-e4b`로 승격된다.

서버나 API가 실패하면 확장은 조용히 브라우저 fallback 결과를 저장하지 않는다. 대신 API 연결 실패를 사용자에게 보여 주고, 최근 작업에도 저장하지 않는다. 실패를 성공처럼 보이게 하지 않는 것이 실제 배포에서는 더 정직한 사용자 경험이다.

이것도 초보 개발자에게 중요한 설계 관점이다.

좋은 도구는 성공 경로만 생각하지 않는다. 실패했을 때 사용자에게 무엇을 숨기고 무엇을 알려야 하는지도 설계해야 한다.

실제 사용 중에는 더 작은 불편도 드러났다. API 키 입력칸이 비밀번호처럼 보이면 브라우저가 저장 여부를 묻기 때문에 일반 텍스트 입력으로 바꾸고 자동완성을 껐다. 결과 요약의 `subject` 칩은 내부 진단에는 useful하지만 일반 사용자에게는 과한 정보라 숨겼다. 긴 `ChatGPT/Gemini에 넣기` 문구는 `자동 입력`으로 줄이고, `복사` 버튼은 초록색으로 구분해 결과 액션이 더 빨리 읽히게 했다.

작업 중인 ChatGPT나 Gemini 창에서 새로고침했을 때 패널이 한 번에 나타나지 않고 두 번째 새로고침에서야 보이는 문제도 여기에서 발견됐다. 원인은 확장 프로그램이 content script 실행 시점에 패널을 한 번만 붙이려고 했기 때문이다. 페이지 shell이 아직 준비되지 않았거나 초기 mount 중 일시 오류가 나면 재시도하지 않고 끝날 수 있었다. 그래서 패널 주입은 실패해도 짧은 간격으로 다시 시도하고, 실패한 host는 정리한 뒤 다음 시도에서 새로 붙이도록 바꾸었다.

Gemini에서 실제로 써 보니 또 다른 문제가 보였다. 페이지는 Gemini인데 패널의 `대상` 값이 `ChatGPT용`으로 남아 있었고, 자동 입력도 실패했다. 콘솔에는 여러 CSP 경고가 있었지만, 우리 확장의 실제 실패는 입력 후 검증 단계에서 난 `verify failed`였다. 그래서 페이지 감지 결과가 Gemini이면 `대상` 선택값도 자동으로 `Gemini용`으로 맞추고, Gemini의 contenteditable 입력창에는 단순 텍스트 대입이 아니라 selection, `beforeinput`, `insertText`, 지연 검증을 함께 쓰도록 바꾸었다.

이후 내부 배포용으로는 API 키도 기본값으로 함께 넣었다. 사용자가 별도로 키를 입력하지 않아도 hosted Prompt API를 바로 호출할 수 있게 하되, 이미 저장한 키가 있으면 그 값을 우선한다.

마지막 점검에서는 예전 로컬 서버 실행 설명과 비어 있는 `EXTENSION_API_KEY` 전제가 문서에 일부 남아 있는 것도 확인했다. 현재 확장 프로그램 폴더에는 Prompt API 서버 소스가 포함되어 있지 않고, 배포판은 hosted Prompt API를 기본 호출한다. 그래서 문서와 개발 명령도 실제 확장 프로그램 배포 구조에 맞게 정리했다.

최종 정리 단계에서는 테스트를 위해 만들었던 `tests` 폴더와 현재 manifest가 쓰지 않는 예전 content script, core 분리 파일, 빈 모듈 폴더를 제거했다. 실행 경로는 `content-runtime.js`와 `background.js` 중심으로 단순해졌고, ZIP에는 구동에 필요한 파일만 남도록 했다.

다음 사용 장면에서는 개선 흐름이 문제로 드러났다. 결과 아래의 `더 선명하게`, `더 한국적으로`, `교육용 안전` 같은 버튼을 누르면 바로 다시 생성이 시작되었고, 사용자는 그것이 재생성 요청인지 모르고 여러 버튼을 연속으로 누를 수 있었다. 그래서 빠른 개선 버튼은 즉시 실행이 아니라 선택 상태로 바꾸었다. 하나를 선택하면 다른 선택은 해제되고, 사용자가 `다시 생성하기`를 눌렀을 때만 API를 다시 호출한다. 입력창 안내도 `수정 방향을 선택하거나 직접 입력한 뒤 다시 생성하기를 누르세요`로 바꾸어 버튼 이름과 흐름을 맞췄다.

진행 상태도 더 눈에 띄게 바꾸었다. 처음 `프롬프트 만들기`를 누르면 버튼 자체가 `프롬프트 생성중...`으로 바뀐다. 개선 생성 중에는 피드백 입력창 안에 `수정사항을 반영하여 다시 프롬프트를 생성중입니다`가 무지개빛으로 표시되고, 완료되면 `수정이 완료되었습니다.`로 바뀐다. 이 작은 변화는 사용자가 지금 무엇이 처리 중인지 알고, 결과가 바뀐 뒤 자연스럽게 위쪽 결과를 확인하게 만든다.

접힌 상태에서도 사용자가 무엇을 여는지 바로 알아야 했다. 그래서 플로팅 버튼은 `프롬프트` 한 단어가 아니라 `프롬프트`와 `만들기`를 두 줄로 보여 주도록 바꾸었다. 첫 사용 안내의 확인 버튼도 그냥 넘어가는 버튼처럼 보이지 않게 `사용방법 확인`으로 바꾸고, 빨간 배경과 흰색 글자로 먼저 누를 행동임을 분명히 했다.

또 API가 내려주는 영어 개선 제안은 그대로 노출하지 않고 한글 안내 문장으로 정리했다. 결과 위쪽에 보이던 `purpose`, `formatPreset`, `description`, `targetSite` 같은 키워드 칩도 일반 사용자에게는 이해 부담이 컸다. 이제 그 영역에는 프롬프트를 한글로 풀어 쓴 설명을 먼저 보여주고, 그 아래에 추가 조언을 붙인다. API가 `promptKo` 또는 `koreanPrompt`를 내려주면 그 값을 쓰고, 없을 때는 확장 프로그램이 프롬프트의 주제와 스타일을 바탕으로 한글 안내를 만든다.

이후 실제 UI를 다시 보니 `작업 유형`에도 스토리북이 있고 `출력 형식`에도 스토리북이 있어 사용자가 같은 결정을 두 번 하는 것처럼 보였다. 그래서 화면에서는 `출력 형식` 목록을 제거하고, 작업 유형 하나가 결과물의 성격과 내부 출력 형식을 함께 결정하게 했다. 이제 사용자는 `스토리북`, `만화/웹툰`, `여러 장 시리즈`, `수업자료 슬라이드`, `A4 문서`, `블로그 대표 이미지` 같은 항목을 작업 유형에서 고르면 된다. 내부적으로는 필요한 `formatPreset` 값을 자동으로 붙여 API 요청을 만든다.

그 다음에는 작업 유형이 단순히 이름만 바꾸는 수준으로는 부족하다는 점이 보였다. 카드뉴스라면 정보 위계와 3-5개 정보 블록이 필요하고, 수업자료 슬라이드라면 교실에서 크게 보이는 개념도와 학습 목표가 필요하다. A4 문서는 인쇄 여백과 섹션 박스가 중요하고, 유튜브 썸네일은 큰 제목 영역과 모바일 가독성이 중요하다. 그래서 작업 유형별 세부 규칙을 프롬프트 생성 과정에 넣었다. 기본 규칙은 자동으로 들어가지만, 고급 옵션의 `추가 조건`에 사용자가 더 원하는 방향을 쓰면 그 요청이 기본 규칙을 구체화하거나 덮어쓸 수 있게 했다.

또 결과를 보는 사람에게도 이 기준이 보여야 했다. 그래서 프롬프트가 생성된 뒤 `프롬프트 한글 설명`에는 단순 요약뿐 아니라 `적용된 작업 유형 규칙`도 함께 표시하도록 했다. 사용자는 결과가 왜 카드뉴스답게, 수업자료답게, A4 문서답게 구성되었는지 바로 확인할 수 있다.

사용 중에는 더 작은 표현 문제도 보였다. 생성이 끝났을 때 잠깐 뜨는 알림은 글자만 작게 보여 놓치기 쉬웠고, OpenAI-compatible 응답이 일반 텍스트로 올 때 LM Studio 같은 내부 구현명이 사용자에게 노출되었다. 그래서 알림은 어두운 배경의 눈에 띄는 박스로 바꾸고, 문구도 `외부 API의 응답을 프롬프트 본문으로 사용했습니다.`처럼 실제 처리 경로를 설명하는 표현으로 바꾸었다.

다시 생성하기에서도 한글 설명이 바뀌지 않는 것처럼 느껴질 수 있었다. API가 `promptKo`를 새로 주지 않으면 확장 프로그램이 자체 설명을 만들었는데, 이 fallback 설명이 수정 요청을 충분히 드러내지 못했기 때문이다. 그래서 개선 요청 내용을 summary에 남기고 한글 설명에도 `수정 요청을 반영했습니다` 문장으로 표시되게 했다. 또한 모델이 생성 장수 조건을 최종 프롬프트에서 빠뜨릴 때를 대비해, 2장 이상 선택된 요청은 정규화 단계에서 `exactly N image prompts` 지시를 최종 프롬프트에 다시 보강하도록 했다.

운영 중에는 timeout 문제도 확인했다. e4b로 승격되는 고품질, 여러 장 생성 요청은 Mac mini와 LM Studio에서 30초를 넘길 수 있었다. 기존 확장 timeout은 이 흐름을 기다리기에 짧았기 때문에, 기본 요청 timeout을 90초로 늘리고 내부 fetch 대기는 95초로 맞췄다. 그래도 시간이 초과되면 단순히 실패라고만 말하지 않고, 서버 응답이 오래 걸리고 있으며 출력 개수나 품질을 낮춰 다시 시도할 수 있다고 안내하도록 바꿨다. 서버 운영 쪽에서는 Nginx와 API 서버 timeout도 120초 이상으로 맞춰야 같은 문제가 반복되지 않는다.

## 9. 티키타카가 만든 품질

이 프로젝트에서 가장 중요한 것은 AI가 코드를 많이 짰다는 사실이 아니다.

중요한 것은 티키타카였다.

사람이 요구했다.

AI가 만들었다.

사람이 확인했다.

문제를 발견했다.

AI가 고쳤다.

다시 사람이 방향을 수정했다.

이 흐름이 여러 번 반복되었다.

예를 들어 스타일 UI도 그랬다.

처음에는 많은 스타일을 보여주는 방향이었다. 그러자 사용성이 떨어졌다. 다시 대표 스타일을 보여주는 방향으로 갔다. 하지만 그것도 “처음 8개”를 보여주는 방식이라 정확하지 않았다. 그래서 “카테고리별 대표 8개”로 수정했다. 이후에는 대표 이미지를 여러 스타일이 공유하는 문제가 보였고, 결국 49개 스타일별 썸네일로 바꾸었다.

스타일 선택 입력칸에는 `Notion LM 50 스타일 보기` 같은 내부 작업 문구가 남아 있었다. 사용자는 그런 이름을 알 필요가 없으므로 제거하고, 다른 목록 입력처럼 선택된 스타일명과 화살표만 보이게 했다.

하지만 스타일을 모두 프리셋으로만 고르게 하는 것도 충분하지 않았다. 어떤 사용자는 준비된 썸네일 중 하나를 고르면 되지만, 어떤 사용자는 “어두운 SF 영화 포스터”, “빈티지 잡지 표지”, “파란 네온 조명과 강한 대비”처럼 자기 머릿속의 스타일을 직접 쓰고 싶어 한다. 그래서 스타일 패널 상단에 직접 입력 칸을 추가했다. 이제 사용자는 49개 스타일 에셋을 참고해 고를 수도 있고, 직접 쓴 스타일을 그대로 API 요청의 `Style` 값으로 보낼 수도 있다.

고급 옵션도 표현을 다듬었다. `이미지 안에 넣을 문구`는 어디에 들어가는지 애매해서 `이미지에 표시할 정확한 문구`로 바꾸었다. 이 칸은 제목, 말풍선, 표지 문구처럼 최종 이미지 안에 실제로 보일 텍스트를 넣는 곳이다. 반대로 `추가 조건`은 이미지 안에 그대로 표시되는 문구가 아니라 스타일, 구도, 금지 요소, 강조점처럼 프롬프트 생성에 추가로 반영할 지시다. 설명만 있으면 사용자가 무엇을 써야 할지 망설일 수 있어서 `예: 배경은 밝은 교실, 텍스트는 최소화, 손가락은 정확하게, 로고/워터마크 제외`처럼 바로 따라 쓸 수 있는 예시문을 placeholder로 넣었다. 참고자료 옵션도 `사용자가 직접 업로드함` 대신 `사용자가 채팅창에 첨부파일로 업로드함`으로 바꾸어, 확장 안에서 업로드한다는 오해를 줄였다.

스토리북과 만화도 그랬다.

처음에는 일반 이미지 생성 중심이었다. 하지만 대화 속에서 교사에게 필요한 실제 작업은 여러 장 구성이라는 점이 드러났다. 그래서 numbered prompt pack 구조가 들어갔다.

가이드 문서도 그랬다.

처음에는 “반영했다”는 말이 추상적이었다. 사용자는 “실제로 프롬프트 생성 때 참고하느냐”고 물었다. 그 질문 덕분에 문서 위치와 코드 반영 위치가 명확해졌다.

API 연결도 그랬다.

처음에는 “로컬 LLM으로 생성하게 해 달라”는 요구에서 출발했다. 그래서 직접 LM Studio 엔드포인트를 열어 두는 방향으로 생각할 수 있었다. 하지만 다시 사용 맥락을 보니, 이 확장은 다른 사람도 불편 없이 써야 했다. 그 판단 때문에 기본 API 주소는 hosted Prompt API로 되돌리고, 로컬 주소 권한은 배포판에서 제거했다.

아이콘도 마찬가지였다.

확장 프로그램을 로드해 보니 이미지가 비어 있었다. 이 발견 이후 image-gen으로 아이콘을 만들고, manifest에 16/32/48/128 크기 아이콘을 연결했다. 작은 누락을 실제 사용 화면에서 발견하고 바로 제품 품질로 연결한 사례다.

결국 티키타카는 단순 대화가 아니었다.

품질 관리 방식이었다.

## 10. 초보 교사 개발자에게 전하고 싶은 메시지

이 프로젝트를 강의로 나눈다면, 저는 세 가지를 강조하고 싶다.

첫째, 개발은 코드보다 생각이 먼저다.

무엇을 만들지보다 왜 만드는지가 먼저다. 사용자가 누구인지, 언제 쓰는지, 무엇이 불편한지 생각해야 한다.

둘째, AI는 정답 기계가 아니라 사고 파트너다.

AI에게 한 번에 완성본을 기대하면 실망할 수 있다. 대신 질문하고, 고치고, 다시 확인하는 파트너로 쓰면 결과가 달라진다.

셋째, Human in the Loop는 선택이 아니라 핵심 역량이다.

AI가 빠르게 만들수록 사람의 판단은 더 중요해진다. 대표 스타일을 고르는 일, 저작권을 걱정하는 일, 수업 상황에 맞게 기능을 바꾸는 일은 사람이 해야 한다.

초보 교사 개발자는 전문 개발자처럼 모든 코드를 외울 필요는 없다.

대신 문제를 잘 정의하고, AI와 잘 대화하고, 결과를 검토하고, 다시 방향을 제시할 수 있어야 한다.

그 능력이 앞으로의 개발 역량이다.

## 11. 마무리: 작은 확장 프로그램이 남긴 것

이 프로젝트의 산출물은 크롬 확장 프로그램 ZIP 파일이다.

하지만 진짜 산출물은 따로 있다.

하나는 교사가 직접 자신의 문제를 제품으로 바꿔 본 경험이다.

또 하나는 AI와 함께 일하는 방식에 대한 감각이다.

이 확장 프로그램은 단순히 이미지 프롬프트를 만들어 주는 도구가 아니다. 교사가 생각을 먼저 정리하고, AI와 대화하며, 수업에 필요한 도구를 직접 만들어 가는 과정의 증거다.

캡스톤 프로젝트는 거창한 기술로 시작하지 않는다.

“수업 준비할 때 이게 불편한데?”

이 작은 불편함에서 시작할 수 있다.

그리고 그 불편함을 붙잡고, AI와 티키타카하며, 직접 검토하고, 다시 고치고, 끝까지 배포 파일로 만들어 내면 된다.

그 과정에서 초보 교사 개발자는 단순 사용자에서 제작자로 넘어간다.

이번 이미지 프롬프트 확장 프로그램은 그 전환을 보여주는 하나의 이야기다.

## 12. 문서도 제품의 일부다

마지막으로 남은 교훈은 문서였다.

코드가 바뀌면 기억도 바뀌어야 한다. 그런데 기억이 사람 머릿속에만 있으면 다음 수정에서 다시 흔들린다.

그래서 이 프로젝트에서는 앞으로 수정이나 변경이 생길 때마다 두 문서를 함께 갱신하기로 했다.

```text
final-spec.md
image-prompt-story.md
```

다른 컴퓨터에서 이어서 작업할 때 대화 흐름을 잃지 않도록 `handoff-context.md`도 만들었다. 이 파일은 새 세션에서 바로 읽을 수 있는 인수인계 메모다. 현재 API 주소, 모델 기본값, 최근 UI 문구 결정, 스타일 썸네일 구조, 검증 명령을 한곳에 모아 둔다.

`final-spec.md`는 현재 제품이 실제로 어떻게 동작하는지 적는 기술 기준서다. API 주소, 모델 순서, 권한, 검증 항목처럼 나중에 다시 확인해야 할 내용을 담는다.

`image-prompt-story.md`는 왜 그렇게 바뀌었는지를 남기는 개발 서사다. 단순히 “무엇을 고쳤다”가 아니라, 어떤 오해가 있었고, 어떤 판단으로 방향을 바꿨는지를 기록한다.

초보 교사 개발자에게 문서는 보고서가 아니라 사고의 흔적이다.

제품이 바뀌면 스펙도 바뀌고, 스토리도 바뀐다. 이 둘을 함께 관리할 때 프로젝트는 단순 코드 묶음이 아니라 설명 가능한 산출물이 된다.

## 강의에서 던질 질문

강의 마지막에는 선생님들께 이런 질문을 던질 수 있다.

- 나는 수업 준비 중 어떤 반복 작업에 시간을 빼앗기고 있는가?
- 그 작업에서 AI가 도와줄 수 있는 부분은 어디인가?
- 그래도 사람이 반드시 판단해야 하는 부분은 무엇인가?
- 내가 AI에게 한 번에 명령하려고 하는가, 아니면 같이 생각하려고 하는가?
- 내 아이디어를 캡스톤 프로젝트처럼 끝까지 산출물로 만들려면 첫 번째 버전은 무엇이면 충분한가?

이 질문에 답할 수 있다면, 이미 개발은 시작된 것이다.
